# Universal Knowledge Model v1

This design document defines the universal ADQ Knowledge Node model. Every piece of content ingested into the platform—whether a Quranic verse, a Hadith, a Fiqh ruling, a historical event, or an astronomical concept—must conform to this structural blueprint. This model ensures that all domains across the platform can seamlessly interoperate, be indexed by the same Search Adapters, and be traversed by the single Relation Engine.

---

## 1. Purpose of the Model
The ADQ Knowledge Model is designed to:
- Standardize the representation of truth across vastly different domains (Textual, Scientific, Historical, Legal).
- Eliminate domain-specific silos.
- Enable frictionless rendering across uniform Reader components.
- Support deep academic research while remaining accessible to beginners.

---

## 2. Separate Canonical Data from Derived Data
To maintain absolute integrity, the knowledge model strictly separates immutable truth from regenerable metadata.

### Canonical Data (Immutable)
This represents the authenticated academic truth. It must never change once published unless correcting a historical error.
- Canonical ID (e.g., `quran:surah:2:ayah:255`)
- Original source text (e.g., Raw Arabic `matn`)
- Source metadata and provenance
- References and primary chains of transmission
- Core structural metadata

### Derived Data (Regenerable)
This represents computed or augmented metadata generated by algorithms, AI, or heuristics. It can be wiped and regenerated at any time as algorithms improve.
- Summaries
- Search tokens and indexes
- Extracted keywords
- Estimated reading time
- Calculated reading difficulty
- AI-generated metadata and embeddings
- Usage statistics and other computed values

---

## 3. Authority Classification
Every node must declare its `authorityClass` to help users understand the origin and theological or academic weight of the information they are viewing.

- `Primary Revelation`: Immutable divine text (Quran).
- `Primary Sunnah`: Authenticated prophetic tradition (Hadith).
- `Scholarly Interpretation`: Human academic consensus or Ikhtilaf (Tafsir, Fiqh).
- `Historical Record`: Documented events and biographies (Seerah).
- `Scientific Observation`: Factual astronomical or geographical data.
- `Educational Material`: Summaries, diagrams, or curated lessons.
- `Platform Generated`: Computed tools, calculators, or AI-derived summaries.

---

## 4. Content Categories Distinction
To further clarify the role of nodes in the graph, it is conceptually vital to distinguish between four major categories:
- **Immutable Primary Sources:** The bedrock of the platform. They provide the ultimate source of truth.
- **Scholarly Interpretations:** Derive their authority strictly by linking back to Immutable Primary Sources.
- **Computed Results:** The output of Islamic Decision Tools (e.g., a Zakat calculation) which dynamically apply Scholarly Interpretation to user input.
- **Educational Content:** Pedagogical hubs that curate the above categories for learning.

*Each category serves a different role while remaining 100% compatible with the universal knowledge model.*

---

## 5. Expanded Provenance
A node must explicitly cite its origin. The `sourceMetadata` object provides a rich provenance model tracing the text back to its archival root:

- `sourceName`: Title of the archive or manuscript (e.g., "Sahih al-Bukhari").
- `edition`: Specific publication or print edition.
- `publisher`: The publishing house or digital archive (e.g., "Dar-us-Salam").
- `translator`: Name of the translator (if localized).
- `language`: Original language of the source.
- `publicationDate`: Historical or print date.
- `license`: Copyright or open-source license.
- `checksum`: Integrity hash of the raw source data.

*(URLs to digital archives are treated as optional metadata, not as primary identifiers.)*

---

## 6. Version Metadata
Versioning is split into distinct domains to trace exactly how and when a node was generated:
- `schemaVersion`: The version of the ADQ Knowledge Node interface structure it follows.
- `contentVersion`: The version of the underlying text (incremented if typos or academic corrections are made to the source data).
- `compilerVersion`: The specific version of the ADQ ingest script that transformed the raw data into this node.

---

## 7. Content Lifecycle Status
Every knowledge node possesses a `status` field to support strict editorial workflows before reaching the user in production.
- `draft`: Initial ingestion, incomplete metadata.
- `reviewing`: Undergoing human QA via Developer Mode.
- `verified`: Approved for public production release.
- `deprecated`: Replaced by a more accurate node (legacy IDs point here).
- `archived`: Removed from active indexing but retained for structural integrity.

---

## 8. Reading Level Metadata
The model formalizes the three audience levels used across ADQ. This metadata dictates which UI components can appropriately render the node based on the user's selected learning level.
- `Beginner`: Accessible, plain-language summaries and practical takeaways.
- `Student`: Intermediate explanations requiring contextual background (Asbab al-Nuzul).
- `Researcher`: Deep academic exploration involving Isnad, original Arabic syntax, and scholarly dispute.

---

## 9. Semantic Tags vs Search Tokens
The model explicitly separates navigation aids from search engine artifacts:
- **Semantic Tags:** Human-readable categories supporting navigation, discovery, and Knowledge Graph clustering (e.g., `Prayer`, `Charity`, `Family`, `Faith`, `Patience`).
- **Search Tokens:** Machine-readable stems, n-grams, and phonetic mappings existing purely for offline fuzzy search indexing.

---

## 10. Node Confidence
A knowledge node may express its academic or historical certainty using a `confidence` metric (e.g., `100%`, `95%`). 
- Immutable primary sources (Quran) inherently carry `100%` certainty.
- Historical reconstructions, weak (Da'if) hadiths, or scientific models may include variable confidence levels, allowing the UI to visually indicate the strength of the evidence to the user.

---

## 11. Purpose Metadata
Every node includes a `purpose` field describing *why* it exists in the graph, enabling better AI recommendations and contextual navigation. Examples include:
- `Teach`
- `Explain`
- `Clarify`
- `Calculate`
- `Preserve`
- `Connect`

---

## 12. Relations and Cross-References
Nodes do not embed their connections directly; they rely on the external Relation Engine.
A relation object comprises:
- `sourceNodeId`
- `targetNodeId`
- `relationType`: `supports`, `explains`, `contradicts`, `mentions`, `chronological_next`.
- `priority`: Visual sorting order.
- `confidence`: Strength of the scholarly link.

---

## 13. Illustrations and Media Attachments
Knowledge nodes can securely reference external media assets (`image`, `audio`, `interactive-canvas`) ensuring rich visual learning always remains tied to canonical evidence.

---

## 14. Extensibility Principles
Future modules might require custom logic, but their definitions and explanations must still map to this node model. If a feature computes a ruling, the explanation of *why* must be a Canonical Node rendered by the standard Reader.
